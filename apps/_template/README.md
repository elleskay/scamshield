# apps/_template

Reference overlay for a real Expo app. These files are not a runnable app; copy what you need into `apps/app/` after scaffolding with `expo prebuild`.

## What's here

- `app.config.ts` - Expo config with the config plugins that wire the native call/SMS modules into the generated `ios/` and `android/` projects.
- `lib/api.ts` - typed API client reading `EXPO_PUBLIC_API_URL`.
- `lib/auth.ts` - access-token storage via `expo-secure-store`.
- `lib/push.ts` - Expo push registration, no-ops without a project id.
- `native/ios/CallDirectory/` - CallKit Call Directory extension (Swift) reference for call blocking/identification.
- `native/ios/MessageFilter/` - `ILMessageFilterExtension` (Swift) reference for SMS filtering.
- `native/android/callscreening/` - `CallScreeningService` + default-SMS role (Kotlin) reference.
- `specs/` - where your `<app>.yml` spec lives.
- `tests/` - jest-expo (unit + component) scaffolding and configs; `.maestro/` holds the e2e flows.

## Native modules: read first

Call blocking and SMS filtering are platform-native app extensions. They cannot be implemented in JavaScript. The JS app manages data (block lists, reports); the OS calls the extensions. See `docs/MOBILE.md` for the full wiring, entitlements, and store-review notes.

The Swift/Kotlin files here are references. A config plugin (`app.config.ts`) copies them into the prebuilt native projects and adds the required entitlements and `Info.plist` / `AndroidManifest.xml` entries. Do not hand-edit `ios/` or `android/`; they are regenerated by `expo prebuild`.
